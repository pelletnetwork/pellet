`@pelletnetwork/cli mcp` runs an MCP (Model Context Protocol) server on stdio. Drop one config line into your agent runtime and the agent gets `pellet_pay`, `pellet_status`, `pellet_balance`, `pellet_recent_events`, and `pellet_lookup_service` as callable tools.

> Same line that today's Stripe Treasury launch shipped: *"Use Treasury from any AI app with the Stripe MCP."* Pellet's MCP is the open / on-chain / self-custody equivalent.

## Pre-requisite

Pair your CLI once (browser passkey ceremony — agents can't drive that step):

```bash
npx -y @pelletnetwork/cli auth start
```

Approve in the browser. After this the agent can spend autonomously within your caps. Re-pair when caps run out or the session expires.

## Install

### Claude Code

Add to `~/.config/claude-code/mcp.json`:

```json
{
  "mcpServers": {
    "pellet": {
      "command": "npx",
      "args": ["-y", "@pelletnetwork/cli", "mcp"]
    }
  }
}
```

Restart Claude Code. The agent now sees `pellet_status` and `pellet_pay` in its tool list.

### Cursor

Settings → MCP → Add Server:

```json
{
  "name": "pellet",
  "command": "npx",
  "args": ["-y", "@pelletnetwork/cli", "mcp"]
}
```

### Cloudflare Agents

Add to `wrangler.toml` agent config (or wherever your runtime expects MCP servers):

```toml
[[mcp.servers]]
name = "pellet"
command = "npx"
args = ["-y", "@pelletnetwork/cli", "mcp"]
```

### Anthropic API direct

Pass to the API as part of your messages request `mcp_servers`:

```json
{
  "mcp_servers": [
    {
      "name": "pellet",
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@pelletnetwork/cli", "mcp"]
    }
  ]
}
```

## Tools

### `pellet_status`

Read the local Pellet Wallet session — caps, expiry, label. Use to confirm wallet capability before attempting a payment.

**Input:** none.

**Output:**
```json
{
  "ok": true,
  "label": "claude-code · daily",
  "spend_cap_usdc": 5.00,
  "per_call_cap_usdc": 1.00,
  "expires_at": "2026-04-30T22:52:35.862Z",
  "paired_at": "2026-04-29T22:52:48.150Z",
  "base_url": "https://pellet.network"
}
```

If no session is paired:
```json
{ "ok": false, "error": "no active session — run `pellet auth start` to pair" }
```

### `pellet_pay`

Sign + broadcast a `transferWithMemo` on Tempo. Uses the agent's on-chain-authorized session key; spend bounded by the user's caps.

**Input:**
- `to` (string, required) — recipient address (0x + 40 hex)
- `amount_usdc` (number) **or** `amount_wei` (string) — required; one or the other
- `memo` (string, optional) — bytes32 hex literal verbatim, or any text → keccak256-hashed to bytes32. For x402 settlement, pass the challenge id.
- `token` (string, optional) — TIP-20 token address. Defaults to chain USDC.e.

**Output (success):**
```json
{
  "ok": true,
  "tx_hash": "0x0af5f25c…",
  "explorer_url": "https://explore.testnet.tempo.xyz/tx/0x0af5f25c…",
  "from": "0x0eBd6a5D…74b5A",
  "to": "0x000…dead",
  "amount_usdc": 0.50,
  "memo": "0x0ae1e216…",
  "spend_used_usdc": 0.50,
  "spend_cap_usdc": 5.00
}
```

**Output (error):**
- `per-call cap exceeded` (server-side reject)
- `lifetime cap exceeded` (server-side reject)
- `session expired`
- `on-chain payment failed: <detail>` (viem / Tempo bubbled error)

### `pellet_balance`

Read the wallet's on-chain USDC.e balance plus the session's remaining cap, in one call. The answer to "can I afford this?" depends on BOTH on-chain balance AND server-side remaining cap, so this surfaces both.

**Input:** none.

**Output:**
```json
{
  "ok": true,
  "managed_address": "0x0eBd6a5D…74b5A",
  "chain": "Moderato",
  "chain_id": 42431,
  "balances": [
    { "symbol": "USDC.e", "address": "0x20c0…3726", "raw_wei": "12345678", "display": "12.345678" },
    { "symbol": "pathUSD", "address": "0x20c0…001a", "raw_wei": "5000000",  "display": "5.000000" }
  ],
  "session": {
    "label": "claude-code · daily",
    "spend_cap_wei": "5000000",
    "spend_used_wei": "600000",
    "remaining_wei": "4400000",
    "per_call_cap_wei": "1000000",
    "expires_at": "2026-04-30T22:52:35.862Z"
  }
}
```

### `pellet_recent_events`

List the user's recent payments. Killer use case: pass `memo_prefix` (an x402 challenge id) to ask **"did I already settle this?"** before re-paying. Closes the agent-side memory loop that every other payment-MCP leaves open.

**Input:**
- `memo_prefix` (string, optional) — case-insensitive challenge-id prefix
- `since` (string, optional) — ISO-8601; defaults to 24h ago
- `limit` (integer, optional, 1–100) — defaults to 25
- `status` (`any` | `submitted` | `pending` | `failed`, optional) — defaults to `submitted`

**Output:**
```json
{
  "ok": true,
  "managed_address": "0x0eBd6a5D…",
  "count": 1,
  "since": "2026-04-30T13:00:00Z",
  "memo_prefix": "0xdeadbeef",
  "events": [
    {
      "id": "…",
      "ts": "2026-04-30T14:21:09.000Z",
      "recipient": "0xabc…123",
      "amount_wei": "100000",
      "amount_usdc": 0.10,
      "challenge_id": "0xdeadbeef…cafe",
      "tx_hash": "0x…",
      "explorer_url": "https://explore.testnet.tempo.xyz/tx/0x…",
      "status": "submitted",
      "reason": null
    }
  ]
}
```

> Memo filtering only matches payments made after the `challenge_id` wiring landed (April 30, 2026). Older rows have null `challenge_id` and won't match `memo_prefix` even if you did pay them.

### `pellet_lookup_service`

Search OLI for a paid x402/MPP endpoint by name, label, or address fragment. Returns watched services with recent activity. Use before paying an unknown service to verify it's real and active. **Public endpoint — no session required.**

**Input:**
- `q` (string, required, min 2 chars) — service slug, label, or address fragment
- `limit` (integer, optional, 1–25) — defaults to 10

**Output:**
```json
{
  "ok": true,
  "count": 1,
  "results": [
    {
      "kind": "service",
      "id": "tempo-gateway-mpp",
      "label": "Tempo MPP Gateway",
      "sub": "0xca4e835f…",
      "href": "/oli/services/tempo-gateway-mpp"
    }
  ]
}
```

## Why pairing isn't an MCP tool

`pellet_auth_start` would have to drive a browser passkey ceremony — but agents can't open browsers, present Touch ID prompts, or pick spend caps on the user's behalf. That step is intentionally human-only. The CLI handles it:

```bash
npx -y @pelletnetwork/cli auth start
```

After that the agent can call `pellet_pay` autonomously, with the caps the user set in the browser.

## Example interaction

```
User: My agent needs to settle a $0.10 x402 challenge to 0xabc…123.
       Challenge id: 0xdeadbeef…cafe.

Agent → pellet_recent_events({ memo_prefix: "0xdeadbeef" })
Tool  → { "ok": true, "count": 0, "events": [] }
        // No prior settlement — safe to pay.

Agent → pellet_balance()
Tool  → {
  "ok": true,
  "balances": [{ "symbol": "USDC.e", "display": "12.34" }],
  "session": { "remaining_wei": "4400000" }
}
        // $4.40 remaining cap, $12.34 on-chain. Both > $0.10. Proceed.

Agent → pellet_pay({
  to: "0xabc…123",
  amount_usdc: 0.10,
  memo: "0xdeadbeef…cafe"
})
Tool  → { "ok": true, "tx_hash": "0x…", "explorer_url": "https://…" }

Agent: Paid. tx 0x… on Tempo testnet.
       Receipt: https://explore.testnet.tempo.xyz/tx/0x…
       Next time the same challenge fires, pellet_recent_events will
       return this row and I'll skip the re-pay.
```
